Durable agents: the core

Repairing anomalous cross-border payments with durable Temporal workflows and Pydantic AI agents.

Objectives

By the end of this session, you'll be able to…

  • Explain why durable execution fits agentic payment repair.

  • Point to exactly where determinism is preserved — and why it must be.
  • Deliver a human decision to a running workflow with a Signal.
Why Temporal here

Agentic payment repair is a durable-execution problem

survive crashes

Every model call runs as a Temporal activity, so an agent's reasoning survives a worker restart and replays deterministically. No in-flight correction is ever lost to a redeploy.

human oversight

A low-confidence correction simply waits — durably, for as long as you allow — for a human decision. No queues, no cron, no lost timers.

no lock-in

The same code runs on any cloud or on Kubernetes. Payloads can be encrypted end to end — a fit for banking's compliance constraints.

Everything you already know from Temporal 101 and 102 is exactly what this domain needs. Three properties matter here. First, agents that survive crashes: because every model call is offloaded to an activity, a worker restart never loses an in-flight correction. Second, human oversight is a first-class, durable state — a workflow can wait for a person for as long as the business needs. Third, it is cloud-agnostic and compliance-ready, which matters when you run payments for banks. We will see all three, live, today.

Step 02

Durable agents & orchestration

Read the baseline: a coordinator, two agent child workflows, the memory-first gate, and determinism.

Now we read the baseline application closely — no feature to enable, just code to understand. This is the coordinator, the two agent child workflows, the memory-first flow, the compliance gate, and the determinism rules that everything else in the workshop builds on.
Step 02 · Baseline

A coordinator, two agents, one gate

instruction agent

A payments-operations expert. Proposes the smallest fix that lets the instruction settle — a valid BIC, a required intermediary bank.

compliance agent

A compliance officer. Does not propose a fix — it returns a verdict on whether a compliant correction is even possible.

the gate

Applies the fix only when the verdict clears it and confidence ≥ 0.75. Anything else is held — fail-closed.

This is the always-on baseline — no feature to enable, just code to read. The heart is a coordinator that fans out to two specialised agents, each running as its own child workflow. The instruction agent proposes the smallest safe fix. The compliance agent does not compete with a rival fix — it returns a verdict. The coordinator treats compliance as a gate, not a proposer: it applies a correction only when compliance clears it and confidence is high enough. A missing clearance never auto-applies. It fails closed and holds for a human.

The coordinator runs both agents as child workflows, concurrently, with gather and return-exceptions set to true. That is deliberate: if one agent fails, the correction degrades gracefully instead of crashing. Both results feed the gate. On APPLY, an activity applies the fix. On REVIEW or no proposal, it holds — fail-closed. In the baseline, a hold just returns applied-false; step 03 turns that hold into a real wait for a human.
Concurrent child workflows

Fan out, then gate on both

# both agents run concurrently, each as its own child workflow
results = await asyncio.gather(
    workflow.execute_child_workflow(InstructionAgentWorkflow.run, anomaly),
    workflow.execute_child_workflow(ComplianceAgentWorkflow.run, anomaly),
    return_exceptions=True,   # one agent failing ≠ total failure
)
proposal, verdict = results   # gather preserves order
The coordinator awaits both agent child workflows at once. return_exceptions=True keeps one agent's failure from taking down the whole correction — the gate then decides on whatever came back.
This is the fan-out in one place. asyncio.gather runs both agent child workflows concurrently. The argument that matters is return-exceptions equals true: if one agent raises, gather returns that exception in place instead of cancelling the other, so a single agent failing degrades gracefully rather than failing the whole correction. And gather preserves order — results zero is always the instruction proposal, results one the compliance verdict — which is exactly what the gate weighs next.

Before either agent spends a model call, it checks corridor memory — a store of known corridor-and-anomaly to correction patterns. If a confident pattern exists, the agent returns it immediately, tagged source equals memory, and never touches the model. That is exactly why the seeded happy path runs with no API key. When the agents do reason out a new fix and it is applied, the coordinator writes the pattern back — but only for model-sourced fixes, never for memory hits, because echoing a memory read would teach nothing.

This is the rule that makes replay safe. Workflow code must be deterministic: no wall-clock, no network, no randomness. Notice the HTTP call to memory does not happen in the workflow — it happens inside the read-corridor-memory activity. All I/O lives in activities or inside the durable agents, which the Pydantic AI integration offloads to activities for you. When you read the code, look for imports-passed-through: that is how the agent objects reach the deterministic sandbox as real objects.

This is the most compelling moment of the session, so slow down. Start a correction that calls the model with the memory-miss scenario — that leaves us a twenty-to-thirty second window. Open the coordinator in the Temporal Web UI and find the pending model activity. Now crash it: press Ctrl-C in the terminal running make dev. The worker, API, and memory service stop — but the Temporal server keeps running, so the workflow stays alive, just with no worker to run the activity. Bring it back with make dev. The worker re-polls, picks up the pending activity, and the correction finishes — it was never restarted from scratch. In the event history, the gap between the activity's scheduled and started times is exactly the outage. The state lived in Temporal, not in the worker.
Step 03

Human-in-the-loop with Signals

Our first feature toggle: turn a held correction into one that waits for a person.

Now we enable our first feature with the toggle CLI. Everything so far was the baseline. In the baseline, a hold is a dead end — the coordinator just returns applied-false. This feature makes "hold for a human" mean actually wait for a human, durably, using a Signal to deliver the decision and a Query to expose the waiting state.
Human approval

When the gate holds, wait for a human

Problem A low-confidence or non-compliant correction must not auto-apply — but "hold" should mean actually wait for a decision, for minutes or for days, surviving restarts.
What Temporal brings The workflow blocks on wait_condition until a Signal delivers the decision; a Query lets clients ask "are you waiting?" without disturbing it.
Observe With the feature on, the coordinator no longer completes — it stays Running, blocked, until you approve.
Speak to this slide, do not read it. The problem first: a held correction must not auto-apply, but holding should mean genuinely waiting for a person — for minutes or for days — and surviving worker restarts the whole time. Temporal makes that a first-class, durable state. The workflow blocks on wait-condition until a Signal — a message you can send from anywhere — delivers the decision. Meanwhile a Query lets a client ask "are you waiting?" without disturbing the workflow. Watch for this in the demo: with the feature enabled, the coordinator stops completing. It stays Running, blocked, until a human decides.

Human-in-the-loop, in code

One signal, one query, one wait

@workflow.signal
async def approve_correction(self, decision) -> None:
    self._decision = decision      # a Signal WRITES, wakes the wait

@workflow.query
def awaiting_approval(self) -> bool:
    return self._awaiting          # a Query READS, no side effect

# in run(): block until a human decides
await workflow.wait_condition(lambda: self._decision is not None)
A Signal writes state and wakes the workflow; a Query reads state with no side effects; wait_condition is where the coordinator durably parks until the decision lands — deliverable from the app or the CLI.
Here is human-in-the-loop as code. The signal handler, approve-correction, is how a decision gets in from outside — it just writes to self-dot-decision. A Signal writes and wakes the workflow. The query, awaiting-approval, reads whether we are blocked, with no side effects — that is what the app's approval panel polls. And the wait itself: wait-condition blocks the run method until the decision arrives. Three decorators, one mental model — Signals write, Queries read, and wait-condition is where the workflow durably parks.

Checkpoint

What we just saw

  • A correction fans out to two agent child workflows, gated fail-closed.
  • An in-flight correction survives a worker crash and resumes on its own.
  • A held correction waits durably; a Signal resumes it, a Query reads it.

All three are the same idea: state lives in Temporal, not in the worker — that is durable execution.

Let's consolidate. Four things, all the same underlying idea. Corrections fan out to child-workflow agents and are gated fail-closed. Memory-first means the seeded path never calls a model. An in-flight correction survives a worker crash and resumes on its own. And a held correction waits durably until a Signal resumes it, while a Query reads its state. Every one of these works because the state lives in Temporal, not in the worker. That is durable execution.
