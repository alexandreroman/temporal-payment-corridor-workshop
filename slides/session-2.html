Step 04

Bounded waiting with durable timers

Stop waiting forever for a human: give the approval wait a deadline that survives restarts.

We start where Session 1 stopped. The human approval wait is correct but unbounded — a reviewer who never answers leaves a correction Running forever. That is an operational risk. This step puts a deadline on the wait using a durable timer, so an unanswered correction auto-rejects instead of blocking indefinitely.
Durable timers

A wait that can end on its own

Problem. "Hold for a human" must not mean "hold forever". A reviewer who never responds should not pin a correction Running indefinitely.
Temporal concept. A durable timer is workflow state, not an in-process sleep. Passing timeout= to wait_condition arms one; it keeps counting across worker restarts and fires deterministically.
Observe. Left unanswered, the correction now auto-rejects with the approval-window message — no background thread, no lost timer.
Speak to this, do not read it. The problem: holding for a human must not mean holding forever. The Temporal concept that fixes it is the durable timer. This is not an in-process sleep that a restart would lose — it is workflow state, just like everything else. We arm it simply by passing a timeout to the wait-condition we already had from Session 1. It counts down across worker restarts and fires deterministically on replay. What to watch for in the demo: leave the correction unanswered and it auto-rejects on its own, with a clear message, once the window elapses.
The change, in one place

Arm the timer, catch the deadline

# _APPROVAL_TIMEOUT: None ⇒ wait forever (baseline)
try:
    await workflow.wait_condition(
        lambda: self._decision is not None,
        timeout=_APPROVAL_TIMEOUT,   # arms a durable timer
    )
except asyncio.TimeoutError:
    # deadline elapsed → auto-reject, survived any restart
    return CorrectionOutcome(applied=False, message=…)
This feature only swaps the timeout constant the Session 1 wait already used — from None to timedelta(minutes=1). The REVIEW branch turns a TimeoutError into an auto-reject outcome.
Here is the entire change, in one place. The wait-condition is the same one we wrote in Session 1 to block on a human decision. All we add is a timeout argument. When the deadline elapses, wait-condition raises asyncio-dot-TimeoutError, and the REVIEW branch turns that into an auto-reject outcome. That is it — the feature literally just swaps the timeout constant from None, meaning wait forever, to one minute. Because the timer is workflow state, a worker restart mid-wait does not reset it.

Step 05

Classifying failures

Learn when not to retry: fail a malformed correction immediately, permanently.

Temporal retries failed activities by default, and that is exactly right for transient failures. But some failures are permanent: a malformed BIC will still be malformed on attempt three. Retrying it wastes time and the retry budget, and delays surfacing the real problem. This step is about telling Temporal the difference.
Retryable vs non-retryable

Decision Retryable path Non-retryable · fail now
A plain exception is retryable by default. Only ApplicationError(non_retryable=True) — or exhausting the policy — stops the retries. Reach for it on validation failures that can never self-heal.
One question decides everything: can a retry ever succeed? If the failure is a bad input — a malformed BIC — the answer is no, and retrying is pointless. That is where you raise an ApplicationError with non-retryable set to true: Temporal fails the activity on attempt one and skips the rest of the policy. If the failure is transient — a network blip, a rate limit — the answer is yes, and you want the default: a plain exception, which is retryable, climbs through the RetryPolicy's three attempts. The key fact to remember: plain exceptions are retryable by default; you opt out explicitly.

Step 06

Reacting to retries

The other side of failure: a transient error Temporal retries — and how to be told it happened.

This is the deliberate counterpart to the previous step. There the error was permanent and retries were wrong. Here the error is transient and retries are right — the correction still succeeds. The problem is that silent retries also hide a rail that is quietly degrading. So we make the activity notice it is being retried and raise an operational alert, while still letting the correction complete.
Retries as a signal

Alert while the correction still succeeds

activity.info().attempt starts at 1 and the server increments it per retry. Cross the threshold and the activity emits corridor_correction_retries_alerted — a climbing counter that can drive a dashboard or an alert rule.
An activity can see its own retry count: activity-dot-info-dot-attempt starts at one and the server bumps it on every retry. Once it crosses a threshold, we emit a custom metric — a counter named corridor-correction-retries-alerted — on the same Prometheus endpoint that already serves the app's metrics. Crucially, the simulated outage is bounded so the correction still succeeds within three attempts, after the alert has fired. So the business outcome is unaffected; you simply get told that a rail is degrading. This is observability driving operations: a climbing counter feeds a dashboard or an alert rule.

Step 07

Long-running activities & heartbeats

Wait for the rail to settle — and meet heartbeats, cancellation, and the replay lesson.

Applying a correction is not the end of the story: the money still has to settle on the downstream rail, which is asynchronous and can take a while. The right shape is a long-running, heartbeating activity that polls the rail. This one step packs three ideas: heartbeats, cancellation, and — because it changes the coordinator's shape — a lesson about replay and versioning. Slow down here; this is the conceptual peak of the session.
Heartbeats

Progress, resumption, and cancellation

Problem. Settlement is asynchronous and slow. The workflow must wait for it — without losing progress on a crash, and without becoming impossible to cancel.
Temporal concept. activity.heartbeat does three jobs: it checkpoints progress, it lets a retry resume from the last checkpoint via heartbeat_details, and it is the channel through which cancellation is delivered.
Observe. From the workflow it is just another activity to await — determinism is preserved. A stalled poll is caught by heartbeat_timeout.
The problem: settlement is asynchronous and slow, so the workflow must wait for it, but a naive wait would lose progress on a crash and be impossible to cancel. The Temporal concept that solves all of this is the heartbeat, and it wears three hats. First, it checkpoints progress. Second, on a retry, the previous attempt's last heartbeat comes back in heartbeat-details, so the activity resumes from the last cycle instead of restarting from zero. Third, it is how cancellation reaches a running activity: the heartbeat call raises CancelledError. And from the workflow's point of view, none of this leaks — it is just another activity to await, so determinism holds. A heartbeat-timeout catches a poll that silently stalls.
Heartbeat = progress + resume

Poll, checkpoint, resume where you stopped

# pseudo-code: resume from the last reported cycle after a retry
cycle = activity.info().heartbeat_details or 0
while cycle < _SETTLEMENT_POLL_CYCLES:
    await asyncio.sleep(_POLL_INTERVAL)   # poll the rail
    cycle += 1
    activity.heartbeat(cycle)            # checkpoint + cancel channel
Reading config and sleeping on wall-clock time are side effects — they live in the activity, never in workflow code. On a retry, heartbeat_details carries the last cycle forward, so the poll resumes instead of restarting.
This is pseudo-code, but it captures the whole shape. The loop polls the rail a fixed number of cycles, sleeping between polls, and heartbeats each completed cycle. Two lines matter. The first reads heartbeat-details from activity-info: on a fresh run it is empty, so we start at zero; on a retry it carries the last cycle we reported, so we resume there. The last line is the heartbeat itself — it checkpoints progress and doubles as the cancellation channel. And notice: reading config and sleeping on wall-clock time are side effects, so they live in the activity, never in the workflow. The workflow just awaits the result.

Here is the lesson that is easy to skip and must not be. Enabling this feature adds a workflow step, which changes the coordinator's Event History. Our replay test replays a committed fixture that was captured with the feature off, so the new code path diverges from the recorded history and replay fails. That failing test is not a regression — it is the lesson, on purpose. In production, the safe way to add a step to already-running workflows is versioning: workflow-dot-patched lets old executions keep their old path while new ones take the new path, so neither diverges from its own history. Step 08

Fleet-wide visibility

Tag every correction with its business dimensions, and query thousands of executions server-side.

Once you run more than a handful of corrections, "show me everything on the US-to-IN corridor" or "show me everything awaiting approval" becomes a real operational need. Temporal answers it without any bespoke query endpoint. This is one of the workshop's headline production concerns: operators get a searchable view of thousands of executions, for free.

The coordinator upserts three typed Search Attributes at the top of its run: corridor, anomaly-type, and a status that carries the lifecycle from processing to awaiting-approval. The call is deterministic and workflow-safe, so it belongs in workflow code. Those fields land in Temporal's Visibility index, and now the whole fleet is filterable server-side — from the CLI and from the Web UI search box — with no bespoke endpoint. One scoping note: this feature only tags the workflows. It deliberately does not touch the payments API or the app, so nothing on the app's screens depends on it.
