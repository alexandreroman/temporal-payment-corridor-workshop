Step 09

Encrypting payloads

Keep bank identifiers and amounts out of plaintext: encrypt at the serialization edge, decrypt on demand.

We open the production session with the headline compliance concern. A cross-border payment carries data you cannot leave sitting in plaintext in Event History — bank identifiers, amounts, beneficiary details. This step encrypts every payload at the edge of serialization, so it travels and rests as ciphertext, and then uses a small codec server to decrypt it on demand for the Temporal Web UI. Encryption on, readability preserved.
Payload codecs

Ciphertext at rest, cleartext on demand

Problem A payment carries data you cannot leave in plaintext in Event History — BICs, amounts, beneficiary details.
What Temporal brings A PayloadCodec sits at the edge of serialization: the SDK calls encode on the way out and decode on the way in, so payloads travel and rest as ciphertext — here with Fernet (AES-128-CBC + HMAC).
Observe Event History now shows ciphertext — until the codec server decrypts it on demand for the Temporal Web UI.
Speak to this, do not read it. The problem: a payment carries data you cannot leave in plaintext in Event History. The Temporal concept that fixes it is the PayloadCodec. It sits at the very edge of serialization — the SDK calls encode on the way out and decode on the way in — so payloads travel the wire and rest in history as ciphertext. Here we use Fernet, which is AES-128 in CBC mode with an HMAC. The trade-off is that Temporal now shows ciphertext too, and that is where the codec server comes in: it reuses the same key to decrypt payloads on demand, so the Web UI stays readable. Two production-minded details in the code: encode tags its output so mixed plaintext and ciphertext histories still decode gracefully, and the Fernet crypto runs off the event loop with asyncio-to-thread.

The codec runs inside your process, so plaintext never leaves it. The codec server reuses the same key to decrypt for display — which is why, in production, it must be authenticated (mTLS or a bearer token) and TLS-terminated, or it becomes an unauthenticated decryption oracle.
Follow the payload. The worker and the API encode it before it leaves the process, so it lands in Event History as ciphertext, and it is decoded back the same way on the way in — that round trip is transparent to your code. The one place ciphertext is a problem is human inspection, so the codec server sits off to the side: Temporal and the CLI call it on demand to display cleartext, using the same key. Say the caveat plainly: because that server decrypts anything you send it, with the insecure dev defaults it is effectively an unauthenticated decryption oracle. In production it must be authenticated and TLS-terminated. The encryption itself, though, happens on your side — even Temporal Cloud would only ever store the ciphertext.

Here is the whole change, in one place, and it is the same in both processes — the payments worker and the payments API. We load the Fernet key from the environment, and if it is missing we refuse to start rather than silently run unencrypted — a deliberate fail-closed choice for a compliance feature. Then the only real edit: we pass a data-converter built around our EncryptionCodec to Client-dot-connect. Read the note about the Pydantic-AI plugin: it only installs its own converter when you do not pass one, so it stays alongside our explicit converter — dropping it breaks the TemporalAgent sandbox validation when the worker starts. Step 10

Durable state as an Entity Workflow

Swap the in-memory corridor store for a single long-lived workflow — durable state with no database behind it.

In the baseline, corridor memory is a plain in-process dictionary — fine for a demo, but it loses everything on restart and keeps no audit trail. This step promotes that state to the Entity Workflow pattern: one long-lived workflow instance that holds the state itself. It also shows off a service hosting its own embedded worker in its own namespace, and it lets us contrast Update against Signal for writes. Watch how little the callers notice.
Entity Workflow

State that lives in the workflow

Problem The baseline corridor memory is a plain in-process dict — it loses everything on restart and keeps no audit trail.
What Temporal brings The Entity Workflow pattern turns that into one long-lived workflow. Reads are Queries (never recorded), writes are validated Updates (durably acknowledged), and continue_as_new keeps history bounded.
Observe The HTTP contract at /api/memory/v1 is unchanged — only the backend behind it moved into Temporal.
The problem: the baseline memory is an in-process dictionary, so it forgets everything on restart and leaves no audit trail. The Temporal concept is the Entity Workflow pattern — a single, long-lived workflow instance whose state lives in the workflow itself, with no disk and no database behind it. Reads are Temporal Queries, which are never recorded in history, so a lookup adds no audit events. Writes are Temporal Updates, and crucially each Update runs a validator before it is admitted to history, so an invalid write is never durably recorded — that is the acknowledged-write advantage of Update over Signal. And because a single workflow's history would grow without bound, run drains its in-flight handlers and calls continue-as-new to keep it bounded while carrying state forward. The payoff: the HTTP contract does not change at all — only the backend behind it moved into Temporal.

A Query read adds no history event; an Update write is admitted only after its @remember.validator passes — so the HTTP 204 is a genuine acknowledgement. The service hosts its own embedded worker in its own memory namespace.
The singleton has a well-known id, corridor-memory, and everything flows through the same HTTP surface as before. A lookup becomes a Query — read only, never written to history — so reads are free of audit noise. A learned pattern becomes an Update, which returns only after its validator passes and the write is durably accepted, so the HTTP two-oh-four the caller gets back is a real acknowledgement, not a hopeful fire-and-forget. And because one workflow accumulating updates forever would blow up its history, once it crosses a maximum it drains in-flight handlers and continues-as-new, keeping history bounded while carrying the accumulated patterns forward. One more thing to point out: the memory service hosts its own embedded worker, in its own memory namespace, and makes startup idempotent across reloads with a use-existing conflict policy.
The change, in one place

The backend moves; the callers don't

# lookup → a Query: read-only, never recorded in history
pattern = await handle.query(
    MemoryWorkflow.lookup, args=[corridor, anomaly_type, bank_id]
)

# remember → an Update: returns only once validated + durably accepted
await handle.execute_update(MemoryWorkflow.remember, pattern)
Both handlers are a REPLACE in memory/app.py: the baseline called store.lookup / store.remember; enabled, they call a Query and an Update. Same routes, same responses — the interface boundary from Session 1 is what makes the swap invisible.
Here is the swap, and it is deliberately small. In the memory service's HTTP handlers, the lookup that used to hit the in-process store now issues a Temporal Query against the singleton, and the remember that used to mutate the dict now issues an Update. Note execute-update: it blocks until the write is validated and durably accepted, which is exactly why the caller can trust the acknowledgement. Everything else — the routes, the request and response shapes — is identical. This is the payoff of putting a clean interface boundary around the memory in Session 1: we replaced the entire backend with a durable workflow and no caller had to change.

The Runtime is built first, so one endpoint serves both families. In production, Temporal Cloud adds a server-side temporal_cloud_* OpenMetrics endpoint and reference Grafana dashboards — no Prometheus to run yourself.
The key wiring detail: the Temporal Runtime, with its Prometheus exporter, is built first, before any client or worker, and the same runtime also backs the application metrics emitted from activities through metric-meter. So one slash-metrics endpoint serves both families side by side: the temporal-star series — task queue, activity, and workflow metrics from the SDK — and the corridor-star series the app defines, like corridor-memory-lookups, corridor-corrections-applied, and the retries-alerted counter from Session 2. In production this local scrape is complemented by Temporal Cloud metrics: a Prometheus-compatible endpoint that exposes server-side temporal-cloud-star series for every namespace, consumable by Datadog or Grafana Cloud, with reference dashboards published to adapt. And one logging discipline worth stating: logging goes through workflow-logger and activity-logger, never print, because the workflow logger is replay-aware.

Step 12

Testing durable code

What's distinctive: replay tests for determinism, and a mocked model so agent tests never touch the network.

The suite ships with the app and covers every package, so the exercise is to run it and read what it asserts. But two things are distinctive about testing durable code, and they are the focus here: the replay test, which guards workflow determinism, and mocking the model so agent tests stay fast, deterministic, and offline. Let us look at the replay test — it is the star.

The test never runs the workflow — it re-executes a recorded history against today's code. Enabling a shape-changing feature intentionally breaks it; in production you reach for versioning, not silent recapture. Agent tests mock the model, so they stay fast, deterministic, and offline.
Temporal recovers a running workflow by replaying its recorded event history against the current code. If a change alters the sequence of decisions the workflow makes — reordering awaited calls, adding one conditionally, changing what a child is passed — replay diverges from history and the instance gets stuck. The replay test catches that whole class of bug at test time. It does not run the workflow: it feeds a previously captured history to the Replayer, which re-executes it against whatever the coordinator looks like now. You saw this in Session 2 — enabling a shape-changing feature intentionally breaks the replay test, because the committed fixture was captured with those features off. That failure is the lesson. In production you would reach for versioning rather than silently recapturing the fixture. Separately, agent tests must never call a real model, so the model is mocked; note the gotcha that Agent-dot-override does not reach the offloaded model activity — you register a TestModel under the real workflow name instead.

To production

Self-host, or point it at Temporal Cloud

no Service to run

Point the same workers and clients at Temporal Cloud — only the address and credentials change. No cluster, datastore, or Visibility store to size, patch, or keep alive.

compliance stays yours

The PayloadCodec encrypts before payloads leave your process, so Cloud only ever stores ciphertext; the codec server that decrypts for the UI runs on your side.

managed namespaces & HA

payments and memory provisioned per-namespace with mTLS / API-key auth; HA namespaces add cross-region failover, no capacity planning.

Throughout the workshop the Temporal Service ran as a dev server in a container. In production you either self-host it — running and scaling the datastore, the front-end, history and matching services, the Visibility store, upgrades and backups yourself — or you point the same workers and clients at Temporal Cloud and operate none of it. For an app like this one, Cloud carries real weight, and three points matter most. First, there is no Service to run: everything connects exactly as it does now, only the address and credentials change. Second, compliance stays yours — the PayloadCodec encrypts before payloads ever leave your process, so even Cloud only stores ciphertext, and the codec server that decrypts for the UI runs on your side, not theirs. Third, the two namespaces this app uses are each managed and isolated, authenticated with mTLS or API keys, and HA namespaces add cross-region failover for the strictest continuity needs — with no capacity planning on your side. You also get the Cloud metrics endpoint and reference dashboards without running Prometheus yourself.
